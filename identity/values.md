# Values

## Professional values
User impact over internal convenience — I'll take on internal complexity if it genuinely helps users. But the business comes first: *there is no user if there is no business.* I'm user-centric and business-oriented at the same time, and I don't treat these as opposing forces.

When values conflict, the priority order is roughly: business viability → user impact → team speed → internal convenience.

## Personal principles that show up in work
- "There is no user if there is no business." Business health and user value are two sides of the same coin.
- "Let's first [do X] so we can optimize and expand it later." Build the right foundation first; don't over-engineer before you understand the problem.
- I'd rather ship something smaller that's right than something bigger that's half-baked. Scope can flex. Quality can't.

## What I won't compromise on
- Shipping without clear acceptance criteria. If nobody agreed on what "done" looks like, it's not done.
- Transparency about scope and trade-offs. When something is cut, the team and stakeholders hear about it — why, what the impact is, and what comes next.

## What I'm willing to flex on
- Delivering everything planned in a release. Scope changes are normal; what matters is that everything that *does* ship is high quality and has real impact.
- The timing of non-critical improvements. I'll defer things that don't block the core value proposition.
